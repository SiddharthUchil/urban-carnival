Executive summary

Scale

We profiled one giant Adobe Analytics table — 3.18 billion rows, 1,198 columns, 357 GB — down to the small slice that is Canada-Retirement web traffic, from two angles: the suite live today and the legacy suite it replaced.

History

The live suite manulifeglobalprod has only 158 days of history (no gaps) — enough to see the strong weekly rhythm (weekends ≈ 40 % of weekdays) but too short to fit yearly/holiday models.

The prize

The legacy suite manugrs recovers ~2.5 years (5.57 M rows) and shows a clean, datable cutover on 2026-02-01 — legacy collapses the exact month the new suite switches on. Splicing them on suite-agnostic metrics lifts the 158-day ceiling.

Filter is wrong

Scope is defined on post_page_url, which is ~37 % blank on the new suite (~48 % legacy). The same filter finds 2–3× more traffic on page_url. Reconfirmed on both suites: scope on coalesce(page_url, post_page_url).

Different tags

The two suites populate different columns — only 12 of ~90 eVars are live in both. A naïve union shows high missing rates; safe cross-suite splices are hits, visits, geography, language — not eVar KPIs.

Clean but anonymous

No bot filtering, ~3-minute batches, Eastern-time stamps, ~0 duplicates — but no person-level identity (cust_visid 100 % null, userid constant). Analysis is device-level, under strict masking.

Blocked on business

Every eVar's meaning is masked (ADR-0007) and the only dictionary on hand is for a different product (CoverMe). Key decisions wait on the GWAM eVar dictionary and a scope sign-off.

01

What we did, in plain terms

One table, two report suites, two notebooks. All Canada-Retirement web analytics land in a single Adobe "hit-level" table (a "hit" is one page view or click). The table mixes every Manulife / John-Hancock property; a report suite (rsid) plus a URL filter is how we carve out just Canada-Retirement.

The two notebooks
NotebookReport suiteWhat it isWhy we profiled it
…_eda.ipynbmanulifeglobalprodNew Storefront site, live todayThe suite the anomaly detector will actually watch.
…_eda_manugrs.ipynbmanugrsLegacy pre-Storefront siteRecover pre-2026 history, and test the manager's challenge that the filter/suite is wrong.

The profiling philosophy. The table is far too large to scan column-by-column, so each notebook does exactly one full-history scan for exact daily volume, then draws a 5 % random sample (seed 42) of a rolling window for every per-column statistic — cheap and representative for shares, with exact counts from the full scans where it matters. Every result is emitted as a machine-readable SHAREABLE JSON block with an integrity manifest, and ADR-0007 privacy is applied throughout: sensitive columns are reported shape-only (how full, how many distinct values, string lengths) — never their values.

The two notebooks are structurally identical; they differ only in the widgets that point them at each suite. A companion charts notebook renders the same data interactively — a visualization layer, not new analysis.

Common misconception

The sample seed is hard-coded (42), the window is derived from window_months (not explicit dates), and there is no SCOPE_URL_MODE / SCOPE_SUITE_MODE toggle in the notebook — those production scope toggles live downstream in the Databricks pipeline (databricks/conf/settings.py), not in the EDA. Explicit start/end dates and geo dropdowns exist only in the charts companion.

03

How we scoped the data

The notebook picks the real column names defensively and builds the scope as lower(trim(rsid_col)) == rsid_filter AND lower(url_col) contains url_filter. Either widget left empty drops that condition; a missing schema column is dropped and flagged, so the notebook never silently profiles the wrong population. Two consequences the audit steps quantify:

  • The URL contains-test silently excludes blank-URL hits. On the new-suite window, post_page_url is blank on 88.2 % of rows — scoping on it alone throws away most of the table before the substring is even tested.
  • The same English filter matches far more traffic on page_url than on post_page_url. The recommended production scope column is coalesce(page_url, post_page_url) — the live notebook still filters post_page_url first, but the evidence (§05) says to switch. Reconfirmed independently on both suites.
04

Which columns we profiled, and how we chose them

The census runs over all 1,198 columns on the 5 % sample and classifies each by how full it is. Adobe writes empty strings, not NULL, so "full" means trimmed non-empty.

Column classes (pop_pct = % non-blank in sample)
ClassRuleMeaning
corepop_pct ≥ 99Reliably populated → usable as a stable time series.
populatedpop_pct ≥ 0.1Carries data, but not always present.
sparse0 < pop_pct < 0.1Almost always empty.
deadpop_pct = 0Never populated in this suite.

Only ~16 % of the 1,198 columns carry any data for Canada-Retirement; the rest are dead for this population. Custom dimensions (evarN/propN) are collapsed to their logical form and counted as "live" (≥0.1 %) and "core" (≥99 %). Dimension candidates for slicing anomalies come from a fixed allowlist filtered to columns that are actually populated.

Privacy (ADR-0007) governs how each column is printed

  • 24 sensitive columns (visitor/device IDs, IPs, fine geo/ZIP, user-agent, account IDs, personalization/social) are reported shape-only — never values, not even masked. A regex net catches any the list misses.
  • A small allowlist of low-cardinality technical dimensions (geo_country, geo_region, browser, os, connection_type, language, hit_source, visit flags) is printed raw.
  • Everything else is masked as <masked:sha1[:8]>; emitted text is additionally scrubbed of emails, IPs, and long hex strings.

S4c — no third suite

A window-wide, rsid-agnostic sweep of strict retirement traffic (7,523,186 rows) splits manugrs 57.30 % + manulifeglobalprod 42.70 % = 99.997 % (residual 195 rows). No third suite carries retirement traffic, and none of the five URL columns holds retirement rows beyond the coalesce (first_hit/visit_start "extras" are visit-entry attribution; site_url is a constant; pagename is ~40 % blank — unusable for scope).

Cross-suite overlap is small: only 12 logical eVars are live in both (evar101–103, 105–109, 131, 138, 162, 200); 50 are legacy-only, 8 are new-only. That is the "different columns populated → high missing rate on a naïve union" case — not a rename. All eVar content is masked (ADR-0007): evar107 = <internal> free-text, evar140 = <hash> (sensitive join-key), evar200 ≈ visitor cardinality. The only dictionary on hand is the CoverMe suite (93 eVars) — a different product, not authoritative here. The 12 shared eVars are the only splice candidates, and only after the business confirms shared meaning.

Events & KPI candidates (S6)

post_event_list is populated on 100 % of hits. New-suite hits carry 16–18 events (p50/p95), max 22; legacy hits carry more, 32/37, max 41 — consistent with the legacy suite's larger eVar set (each live eVar fires a presence flag). Event IDs decode by the standard Adobe convention (not a suite dictionary): 20 = Campaign View, 500–504 = clickmap, and 10000+k = "Instance of eVar(101+k)". These instance events fire but carry no numeric value — presence flags. No custom business events (200–1000) fire in either suite.

Instrumentation change-points (new suite)

ev501–504 start 2026-02-24; ev20 starts 2026-03-03; ev500 fires only 2026-04-02 → 06-15 then stops. These are tag changes — the detector must treat them as known change-points, not alerts.

On the notebook's event labels

The notebook has no custom-event lookup loaded, so IDs print as "unknown" (0 of 23 resolved on the new-suite re-run, 0 of 40 on the legacy). That is a display gap, not a data gap — the decode convention above is authoritative.

Data quality & identity (S10, S11)

Uniformly clean on both suites: key-column null/blank ≤ 0.024 %; zero server-side bot filtering (exclude_hit × hit_source = (0,1) for 100 % of hits, so clean_hits == hits — bots are not removed, we own that heuristic); clock skew −5h/−4h, so date_time is Eastern; duplicate rate ≈ 0.

No person-level identity

cust_visid / post_cust_visid are 100 % null; userid / username / cookies are constant. Visitors are approximated device-level from mcvisid (~50 K distinct). Under ADR-0007, device IDs are pseudonymized (keyed HMAC-SHA-256, key-versioned), IPs dropped, fine geo generalized. Pseudonymized data is still personal information under Law 25 / PIPEDA — a PIA is required before production ingestion; stitching is deferred pending a MarTech ask.

The legacy suite ran healthy from Jan 2024 → Jan 2026 (~100–520 K hits/month) and collapsed in Feb 2026 exactly as the new suite switched on — one population handed to another, not two overlapping feeds. The production pipeline already carries the switches to act on this — a SCOPE_URL_MODE (en_only vs broad) and a SCOPE_SUITE_MODE (current_only vs with_legacy) in databricks/conf/settings.py and databricks/src/01_bronze_ingest.py — held off until the business signs off scope, because flipping either re-baselines every KPI.

On alert thresholds

The EDA sets the baseline shape (DOW-aware, change-point-marked); it does not set alert magnitudes. The level-shift sensitivity figure (×1.9) used elsewhere comes from the separate detectability-floor analysis in the detect/ engine — not from this EDA.

08

Open questions for business / analytics owners

Scope & history
  1. Confirm 2026-02-01 as the migration date and whether manugrs history is authoritative for backfill. Union the legacy suite for training (≈2.5 yr, at the cost of a change-point) or start clean at the new suite (158 days)?
  2. Should scope include French /ca/fr traffic? It is now a two-domain decision (legacy +314 K rows, new +230 K rows).
Metric semantics
  1. Provide the GWAM eVar dictionary for both suites — content is masked and the CoverMe dictionary is not authoritative. Priority: evar107 (free-text — PII-safe?) and evar140 (sensitive hash/join-key).
  2. For the 12 shared eVars, does each capture the same dimension in both suites? Only these can be spliced.
  3. Which KPIs should actually trigger alerts (enrolment funnel? campaign views? traffic? language mix?)?
Dimensions & operations
  1. Is the new suite's USA 11.5 % real end-users or internal/test traffic to exclude?
  2. Confirm the language mapping (45 = EN, 39 = FR).
  3. Freshness SLA / daily cutoff (writes are ~3-min batches; grain of record is daily).
  4. Is weekend/holiday ≈ 40 % expected, and should statutory holidays be modelled?
