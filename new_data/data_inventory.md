# GMAI-Pulse — `data/` Folder Inventory

> **Purpose.** Decision-oriented discovery report for the files currently sitting under `data/`. Intended for architects and data scientists solutioning the GMAI-Pulse **Anomaly Detection & Insights Agent**. Answers: what each file is, what it means in business terms, and how much detection / explanation / enrichment value it carries for Phase 1.
>
> **Companion artifact.** An automated profile generated by [src/gmai_pulse/profiling/data_profiler.py](../../src/gmai_pulse/profiling/data_profiler.py) is stored alongside this document at [docs/discovery/generated_data_profile.md](generated_data_profile.md) and [docs/discovery/generated_data_profile.json](generated_data_profile.json). ## Data-handling & safety

- Inspection was **fully local** to the repository; no external APIs, no uploads, no network egress.
- This report contains only **aggregate statistics, column names, data types, and masked/hashed samples**. No raw hit rows, no PII, no internal URLs from the source workbooks, no credentials.
- Columns flagged sensitive by the profiler's heuristic (`ip`, `ipv6`, `cookies`, `geo_zip`, visitor-cookie fields, `post_tnt`, etc.) are named for schema completeness but their values are never displayed here — the profiler masks them with SHA1 truncation, and this document additionally omits even those masked forms.
- Sensitive-column detection is **heuristic, not a compliance audit.** A separate data-classification review is a prerequisite before any production ingestion of the sample file.

### What it actually is

Despite the file name, this is **not** a retirement product dataset. The sheet shape (1,198 columns, columns named `post_evar1..N`, `post_prop1..N`, `post_event_list`, `visid_high/low`, `hit_time_gmt`, `geo_*`, `user_agent`, `ip`, `ipv6`, `cookies`, `browser`, `browser_width`, `first_hit_referrer`, `visit_referrer`, etc.) matches the **canonical Adobe Analytics data-feed hit-level schema** exactly. The sheet is literally named `result`, strongly suggesting it was exported from an ad-hoc query (Adobe Analytics data warehouse, or a Snowflake/Databricks query against a landed data feed) filtered to a "Canada Retirement" report suite, segment, or product line — with the *query name* preserved as the file name.

**Best inference:** a 10-row hit-level snippet from the CoverMe Adobe implementation, restricted to Canada-region retirement-product traffic (likely Manulife Canada's retirement/insurance funnel that CoverMe sits within). Purpose in this repo: **prove the feed shape** so GMAI-Pulse solutioning can reason about which columns will actually carry signal.

### Column categorization (populated columns only)

Only 26 of 1,198 columns hold any values in this sample. Grouped by Adobe data-feed convention:

| Group | Column(s) | Purpose in Adobe |
|---|---|---|
| **Campaign / attribution** | `campaign`, `post_campaign` | Tracking-code / campaign parameter (raw + post-processed) |
| **Referrers** | `referrer`, `post_referrer`, `ref_domain`, `visit_referrer`, `visit_ref_domain`, `first_hit_referrer`, `first_hit_ref_domain` | Traffic-source attribution at hit / visit / first-hit granularity |
| **Landing / first hit** | `first_hit_page_url`, `first_hit_pagename` | First page of the visit |
| **Keywords** | `visit_keywords`, `post_keywords` | Search keywords associated with the visit |
| **eVars in use** | `evar140`, `post_evar140`, `post_evar193`, `post_evar194` | Conversion variables — business meaning comes from the dictionary (File 2, `post_eVar`) |
| **Props in use** | `prop55`, `post_prop55` | Traffic properties — meaning from `post_prop` sheet |
| **Click-map** | `post_clickmaplink`, `post_clickmaplinkbyregion`, `post_clickmappage`, `post_clickmapregion` | Adobe ClickMap analysis fields |
| **Tag / integration signals** | `ch_hdr`, `ch_js`, `post_tnt` | `ch_hdr`/`ch_js` appear to be channel-detection markers from a launch/tagging library; `post_tnt` is the Adobe Target integration ID (flagged sensitive by heuristic) |

**Categorization by GMAI-Pulse taxonomy (metrics / events / dimensions / mappings / metadata / logs / free-text):** this file is primarily **dimensions + events**, with **timestamps latent in the wider schema** (`hit_time_gmt`, `cust_hit_time_gmt`, `date_time`, `first_hit_time_gmt`, `visit_start_time_gmt` are declared in the dictionary but not populated in this 10-row slice). No standalone metric columns are populated — Adobe metrics live inside `post_event_list` (also empty in this sample).

**Profiler heuristic caveat.** The `is_timestamp` and `is_metric` flags on wide-integer Adobe columns like `browser`, `browser_height`, `country`, `color`, `duplicate_events`, `mobile*` should be **ignored** — they trip a name-pattern match but are actually Adobe numeric ID codes (looked up via the `browser.tsv`, `country.tsv` etc. lookup files that ship with a real data feed). This is a documentation-side note, not a bug in the sample.

### Sensitive columns (names only — values never displayed)

The profiler flagged 24 columns as potentially sensitive. All are legitimate Adobe hit-level identifiers that any downstream pipeline **must** hash, tokenize, or drop before storage:

- **Network / device identifiers:** `ip`, `ip2`, `ipv6`, `j_jscript`, `javascript`
- **Cookie / visitor identifiers:** `cookies`, `post_cookies`, `persistent_cookie`, `post_persistent_cookie`, `daily_visitor`, `hourly_visitor`, `monthly_visitor`, `yearly_visitor`, `new_visit`, `visid_high`, `visid_low`, `visid_new`, `visid_timestamp`, `visid_type`
- **Geo precision:** `geo_zip`
- **Third-party integration IDs:** `post_tnt` (Adobe Target)
- **Social / account fields:** `post_socialaccountandappids`

None of these values are pasted anywhere in this document. Whether they are actually populated in this specific 10-row sample is not disclosed in this report — assume worst-case (populated) for classification purposes.

### Data-quality observations

- **Extreme sparsity by design.** 972 of 1,198 columns are 100% null. This is normal for Adobe data feeds — the schema declares slots for all possible eVars/props/events regardless of whether the property uses them. It is *not* a data-quality problem; it is a signal that CoverMe uses ~2 % of the available Adobe schema.
- **10 rows = discovery sample, not a training corpus.** No time-series analysis, seasonality modelling, or baseline estimation is possible from 10 rows. Treat this file exclusively as a **schema-confirming artifact**.
- **No duplicate rows.**
- **Timestamps not visible in the sample.** The declared hit-time columns (`hit_time_gmt`, `date_time`, `cust_hit_time_gmt`, `first_hit_time_gmt`, `visit_start_time_gmt`) are among the 972 empty columns. That's a puzzle — a real hit-level export should always carry at least `hit_time_gmt`. Either the export truncated timestamp columns, or the sample was hand-curated for privacy. Worth confirming with the source of the extract.

### Business meaning

**Web analytics — Adobe hit-level sample for the CoverMe Canadian retirement funnel.** Value here is *structural*, not *statistical*: it confirms the pipeline shape GMAI-Pulse will need to consume (Adobe data-feed TSV/CSV with a stable ~1,200-column schema, per-hit granularity, per-visit and per-first-hit dimensions materialized). Ten rows cannot drive detection; a rolling window of production hits can.

| Property | Value |
|---|---|
| Rows × Columns | 224 × 4 |
| Columns | `Column name`, `Description`, `Data type`, `Notes` |
| Populated columns | 3 of 4 (`Notes` is 96 % null) |

**What it is.** The authoritative list of the **224 base columns** in the CoverMe Adobe data feed, with SQL-flavoured types (`varchar(255)` × 97, `varchar(100)` × 23, `tinyint unsigned` × 23, `char(20)`, `text`, `int`, `bigint`, …) and a free-text `Description` field per column.

**Why 224 columns here vs. 1,198 in the sample.** The dictionary describes the *base* Adobe schema. In the raw feed, `post_evar1..N`, `post_prop1..M`, and `post_event_list` slot expansions balloon the column count to ~1,200. The dictionary handles those via the sibling sheets below.

**Notable `Notes` content.** ~9 rows carry `"Anomaly Detection"` in `Notes` — a **direct business signal** that these specific columns are pre-approved detection targets. This list should be lifted verbatim into the GMAI-Pulse metric registry.

**Sensitive flag.** The profiler flagged `Description` as sensitive on a value-heuristic (long free text triggered a false positive). It is not — this is a documentation field.

### Sheet 2.2 — `post_eVar`

| Property | Value |
|---|---|
| Rows × Columns | 93 × 4 |
| Columns | `Column`, `Name`, `Status`, `Notes` |
| Populated columns | 3 of 4 (`Notes` 91 % null) |
| Content type | Mappings (eVar-slot → business label) |

**What it is.** The **eVar catalog** — one row per `post_evar1..post_evar93` slot with:

- `Column` — the raw feed column name (e.g., `post_evar1`, `post_evar140`).
- `Name` — the business-facing label (e.g., `Time Stamp`, `New/Repeat Visitors`, `Visit Number`, `Language`).
- `Status` — all `Enabled` in this snapshot.
- `Notes` — ~8 rows carry `"Anomaly Detection"`.

**Value for GMAI-Pulse.** This is the **primary enrichment table for the sample file's populated eVars** (`evar140`, `post_evar140`, `post_evar193`, `post_evar194`). Any anomaly on a raw `post_evarN` column can be re-labelled with the human-readable name straight from this sheet — a Day-1 UX win for the Insights Agent output.

**Data-quality nit.** 92 distinct names for 93 slots — `Language` is used twice. Confirm whether this is intentional (e.g., page language vs. content language) or a legacy duplicate before treating `Name` as a unique join key.

| Property | Value |

| Columns | `Column`, `Name`, `Status` |
| Populated columns | 3 of 3 |

**What it is.** The **prop (traffic-variable) catalog** for `post_prop1..post_prop26`, all `Enabled`. Business labels observed include product-oriented dimensions like `Product Category`, `Product ID`, `Sponsor/Distributor/Association`, etc.

### Sheet 2.4 — `post_event_list`

| Property | Value |
|---|---|
| Rows × Columns | 156 × 5 |
| Columns | `post_event_list number`, `Event`, `Friendly Name`, `Status`, `Notes` |
| Populated columns | 3 of 5 (`Friendly Name` 6 % null, `Notes` 92 % null) |
| Content type | Mappings + metadata (event-ID → business event + monitoring flag) |

**What it is.** The **event catalog** — 156 numbered Adobe events with their machine `Event` name (`Purchase`, `Product View`, `Cart Open`, `Cart Add`, `Cart Remove`, …), a human `Friendly Name`, and a `Status`.

**High-value observations.**

1. **~12 events carry `"Anomaly Detection"` in `Notes`** — the largest concentration of pre-declared detection targets in the corpus. These are effectively the **conversion / commerce KPIs the business wants alarms on**. Extract them first.
2. **Case-inconsistency data-quality nit.** `Status` has two values: `Enabled` (93 rows) and `ENABLED` (55 rows). Same semantic, different casing. Any code that groups on status must lower-case first. Small, but worth flagging — inconsistent casing in catalog data is often a symptom of catalog rows being appended by different tools over time.
3. **~10 events have no `Friendly Name`.** Those will render as raw event codes in Agent output unless a fallback label is generated.

**Value for GMAI-Pulse.** This is the **event-name enrichment table** for anything derived from `post_event_list` in the hit feed. Also the **strongest candidate signal source in the whole corpus** for Phase-1 detection targets, because commerce events (`Purchase`, `Cart Open`, `Product View`) are exactly the KPIs stakeholders in *Demand Generation* and *Measurement & Insights* care about (see [README.md#scope--stakeholders](../../README.md#scope--stakeholders)).

## Anomaly-detection value assessment

### Promising for detection (files carrying candidate signals)

| File | Verdict | Rationale |
|---|---|---|
| [data/Canada Retirement.xlsx](../../data/Canada%20Retirement.xlsx) | **Structurally promising, volumetrically insufficient.** | Confirms CoverMe emits Adobe hit-level data with usable dimensions (campaign, referrers, product eVars/props, click-map, page URLs) and event slots. But 10 rows cannot drive any statistical baseline, seasonality model, or KPI-aware threshold. **Cannot be used directly for Phase 1 detection.** |
| Dictionary sheets | Not signal-bearing themselves. | Metadata only; they define *what* to detect on, not *values to detect from*. |

**Critical gap for Phase 1 solutioning.** The current corpus contains **no time-series data**. Every solutioning conversation about detection approach (statistical vs. ML vs. hybrid, per-metric tuning, seasonality handling) is currently un-groundable in real data. **Sourcing a real Adobe hit-level feed for CoverMe covering ≥30–90 days is the single highest-leverage next step.**

### Explanatory (files that could explain / correlate with anomalies)

**None present.** The corpus contains **no deployment logs, no tagging-change history, no release notes, no campaign launch calendar, no CMS change feed**. The Insights Agent's root-cause-hint capability (a core pillar per [README.md#proposed-solution](../../README.md#proposed-solution)) has no evidence base to draw from yet.

Sources that would fill this gap:

- Adobe Launch / tag-manager change history (who edited which tag, when).
- CoverMe web deployment history (release timestamps, per-page or per-component).
- Marketing campaign calendar (campaign launches, spend changes, creative rotations).
- CMS content-change log (major page or template updates).
- Consent-banner / CMP configuration change history.

### Enrichment (files that could add context / labels to anomalies)

| Enrichment kind | Source in the corpus | Value |
|---|---|---|
| **eVar business labels** | `CoverMe … Data Dictionary … Web - Data Feed.xlsx#post_eVar` | High — decode `post_evarN` → business name (e.g., `post_evar140` → its declared purpose). |
| **Prop business labels** | Same file, sheet `post_prop` | High — decode `post_propN` → product-oriented dimension names. |
| **Event business labels** | Same file, sheet `post_event_list` | Highest — decode `event42` → `Purchase`, etc. Ready-made human labels for Agent output. |
| **Column descriptions & types** | Same file, sheet `data_feed_columns` | Medium — useful for schema-driven validators and for the Agent to explain what a column means when producing insights. |
| **Pre-declared detection targets** | The 29 rows tagged `"Anomaly Detection"` in the `Notes` columns across the dictionary sheets | **Highest strategic value** — a business-approved starter list of metrics to monitor. Extract into a canonical, versioned catalog file. |

### Missing enrichers that would materially improve output

- **URL → owner mapping** (which team owns which page / product / funnel step).
- **Region / brand → report-suite mapping** (helps carve out per-region baselines).
- **KPI → owner mapping** (routing anomalies to the right person, per the "Routing & workflow" pillar).
- **Site inventory** (list of pages that should exist and be reachable, for coverage anomalies).

## Appendix B — How to regenerate the machine profile

From the repo root, with the project virtualenv active and `pandas` + `openpyxl` installed:

```bash
python src/gmai_pulse/profiling/data_profiler.py \
    --input data \
    --output docs/discovery \
    --sample-rows 50000 \
    --verbose
```

Outputs (all under [docs/discovery/](.)):

- [generated_data_profile.md](generated_data_profile.md) — aggregate human-readable inventory table.
- [generated_data_profile.json](generated_data_profile.json) — aggregate machine-readable profile.
- [profile_outputs/](profile_outputs) — one JSON per source file (Excel = list of per-sheet profiles inside one JSON).

The profiler is offline-only, masks sensitive column values with SHA1 truncation, samples the first 50,000 rows per file (all rows here — every sheet has < 250), and is safe to re-run (`--resume` is default; `--force` re-profiles everything).
